# RESOLVER.md

Routing table for filing new content and loading context. Machine-managed region regenerated by `/build:build-resolver`; human prose outside markers is preserved.

<!-- resolver:begin -->
## Filing — where new content goes

| Content type | Location | Naming |
|---|---|---|
| Project context doc (domain knowledge, patterns, conventions) | `.context/` | `<slug>.context.md` |
| Implementation plan for a feature or change | `.plans/` | `YYYY-MM-DD-<slug>.plan.md` |
| Design doc / RFC / architectural sketch | `.designs/` | `YYYY-MM-DD-<slug>.design.md` |
| Research investigation (SIFT framework) | `.research/` | `YYYY-MM-DD-<slug>.research.md` |
| Reusable prompt (skill-dev, maintenance, automation) | `.prompts/` | `<slug>.md` |

## Context — what to load before acting

| When doing… | Load first |
|---|---|
| Authoring or auditing a primitive pair | `plugins/build/_shared/references/skill-pair-best-practices.md`, `plugins/build/_shared/references/primitive-routing.md`, `plugins/build/_shared/references/` (browse for the relevant `<primitive>-best-practices.md`) |
| Authoring a Claude Code skill | `plugins/build/_shared/references/skill-best-practices.md`, `plugins/build/_shared/references/primitive-routing.md` |
| Planning a feature | `plugins/work/_shared/references/plan-format.md`, `plugins/work/_shared/references/feedback-loop.md` |
| Running SIFT research | `plugins/wiki/skills/research/references/` |
| Writing a resolver | `plugins/build/_shared/references/resolver-best-practices.md`, `plugins/build/skills/check-resolver/references/audit-dimensions.md` |

## Out of scope

- `.claude/` — local Claude Code config; not filing target
- `.claude-plugin/` — plugin marketplace manifest; not filing target
- `.github/` — CI config; not filing target
- `.pytest_cache/`, `.ruff_cache/` — tool caches; ephemeral
- `plugins/` — code, not filing; has its own per-skill structure
<!-- resolver:end -->

## Notes

Add context bundles or filing conventions here; prose outside the managed region survives regeneration.

When filing into a directory not listed above (and not in **Out of scope**), run `/build:check-resolver` first. It either confirms the directory is a dark capability that needs a filing row, or surfaces drift to regenerate. Routing follows the most specific `RESOLVER.md` on the filing path — today the root resolver is the only one, but nested resolvers may be introduced for directories with rich substructure.
