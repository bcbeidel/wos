Keep bash scripts under 300 non-blank lines. When a script grows past the cap, extract cohesive sections into helper files or convert to a real language.

**Why:** bash's lack of data structures (no proper maps/sets, awkward arrays), weak error handling (only `set -e` and trap-based propagation), and weak abstraction (functions are flat namespaces, no modules) does not scale. Past ~300 lines, the cost of maintaining the script — debugging, refactoring, onboarding the next maintainer — exceeds the cost of either splitting it into smaller scripts or rewriting it in a language with proper data types and exception handling. The 300-line threshold is empirical: scripts smaller than this stay tractable; scripts larger than this become hairballs.

**How to apply:** when the audit fires, evaluate two paths:
- **Extract** cohesive sections into helper scripts (`source`d from a top-level orchestrator). Useful when the script has clear phases (fetch → transform → validate → write) that can become `_lib_fetch.sh`, `_lib_transform.sh`, etc.
- **Convert** to Python (or another language) when the logic genuinely outgrew bash — when you're emulating data structures, when error handling is fragile, when string parsing dominates. `/build:build-python-script` is the next stop.

```bash
# Top-level orchestrator after extraction
#!/usr/bin/env bash
set -euo pipefail

source "$(dirname "${BASH_SOURCE[0]}")/_lib_fetch.sh"
source "$(dirname "${BASH_SOURCE[0]}")/_lib_transform.sh"
source "$(dirname "${BASH_SOURCE[0]}")/_lib_validate.sh"

main() {
  local raw transformed validated
  raw="$(fetch "$@")"
  transformed="$(transform "$raw")"
  validated="$(validate "$transformed")"
  printf '%s' "$validated"
}

[[ "${BASH_SOURCE[0]}" == "${0}" ]] && main "$@"
```

**Exception:** generated bash scripts (e.g., a CI step that produces a deployment script) may legitimately exceed the cap because they're not human-edited. Document the choice in a header comment ("generated by X; do not edit by hand").
