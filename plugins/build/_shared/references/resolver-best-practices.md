# Resolver Best Practices

## What a Good Resolver Does

A resolver is a routing table for dynamic context. Two tables live in one markdown file at the repo root: **filing** (when Claude writes new content, where does it go?) and **context** (before Claude acts on a recurring task, which docs does it load?). The purpose is to replace bloated always-on context with pointers the model walks on demand — the documents themselves load only when the decision tree says to load them.

A resolver earns its place when a repo has enough tracked directories with filing conventions, or enough recurring tasks with known reference-doc sets, that asking Claude to rediscover them every session wastes context budget. Below that threshold, AGENTS.md and skill-level `references:` frontmatter are sufficient.

Scope is narrow: information routing only. Skill-dispatch routing (task → which skill runs) is handled by the `description` field on each SKILL.md and is not the resolver's concern.

## Anatomy

Three linked artifacts, one responsibility each.

**`RESOLVER.md`** at repo root (sibling to `AGENTS.md`):

```markdown
# RESOLVER.md

Routing table for filing new content and loading context. Machine-managed region regenerated by `/build:build-resolver`; human prose outside markers is preserved.

<!-- resolver:begin -->
## Filing — where new content goes

| Content type | Location | Naming |
|---|---|---|
| research | `.research/` | `YYYY-MM-DD-<slug>.research.md` |
| design | `.designs/` | `YYYY-MM-DD-<slug>.design.md` |
| plan | `.plans/` | `YYYY-MM-DD-<slug>.plan.md` |
| scoped context | `.context/` | `<slug>.context.md` |
| shared reference | `plugins/<plugin>/_shared/references/` | `<slug>.md` |

## Context — what to look at before acting

| When doing… | Look in |
|---|---|
| planning research | `.research/` |
| planning work | `.plans/_index.md` |
| authoring a rule | `_shared/references/rule-best-practices.md` |

## Out of scope

- `.raw/` — free-form ingress, no filing rule
- `.cache/` — ephemeral
<!-- resolver:end -->

## Notes

<free-form human prose outside managed region — stays across regenerations>
```

**`AGENTS.md` pointer** — one line added (not auto-managed):

> Before filing new content or loading context beyond a skill's eager `references:`, consult `RESOLVER.md`.

**`.resolver/evals.yml`** — trigger evals sidecar:

```yaml
cases:
  - prompt: "save this investigation as research on agentic resolvers"
    expected_filing: .research/
  - prompt: "I want to plan the next refactor"
    expected_context: [.plans/_index.md]
  - prompt: "dump this stripe webhook payload for later"
    expected_filing: .raw/
  - prompt: "authoring a new rule for staging models"
    expected_context: [_shared/references/rule-best-practices.md]
```

## Authoring Principles

**Two tables, one file.** Filing and context share one document because they route the same underlying question — "what's the canonical doc for X?" — in opposite directions (where to write vs. what to read). Splitting them hides one behind the other and invites drift.

**Root placement, sibling to AGENTS.md.** The resolver must be reachable from the root of the context or the model can't find it. A nested resolver inside `plugins/` or `.docs/` is invisible to a first-turn task because AGENTS.md is the only file loaded unconditionally.

**Machine-managed region, human context.** The filing table is generated from disk conventions (directories with `_index.md`, frontmatter `type:` fields, naming patterns). The managed region is bounded by `<!-- resolver:begin -->` / `<!-- resolver:end -->` markers matching the wiki-managed pattern already in use. Prose outside the markers survives regeneration; edits inside are stomped and flagged by the auditor.

**Disk-derived, not hand-curated.** Hand-editing the filing table is the failure mode the check catches. The filing table reflects observable reality (what directories exist, what their `_index.md` says); the check flags drift in both directions — tables that lie about the repo and repos that grew directories the table doesn't know about.

**Cross-link, don't restate.** Filing rows point at `_index.md` files and directory conventions; they don't copy contents. Context rows point at specific files to read or directories to look in (the agent consults `_index.md` first, descending on need); they don't summarize the docs. Duplication produces drift on the first rename and rewards maintenance effort in the wrong place.

**Primary-subject filing.** Content is filed by its primary subject, not by source format, producing skill, or incidental metadata. A Slack thread about a person goes to `.people/`, not `.inbox/`; a research doc about a pipeline goes to `.research/`, not `.pipelines/`. The filing row's column order — *content type first, location second* — enforces this.

**AGENTS.md pointer, not skill mandates.** One line in AGENTS.md anchors the resolver. No per-skill preambles are injected. Skills stay untouched; the resolver becomes discoverable via the AGENTS.md trail that Claude reads at the start of relevant interactions. If the pointer is missing, the resolver is unreachable — that's the auditor's first assertion.

**No runtime indirection beyond the pointer.** Claude reads `RESOLVER.md` as a normal markdown file via the AGENTS.md link. No custom loader, no slug-resolution layer, no frontmatter innovation beyond what Anthropic already documents. The mechanism is deliberately boring.

**Trigger evals prove routing.** ~10–30 seeded prompts mapped to expected outcomes (filing location for writes, loaded doc set for reads). A resolver that cannot prove it routes correctly is a decoration, not an enforcement. Evals run on demand, in CI, or against the managed region after every regeneration.

**Reachability + staleness as signals, not gates.** Every directory on disk is either in the filing table or explicitly marked out-of-scope. Dark capabilities — tracked directories the resolver doesn't know about — are the "surgeon the hospital can't find" failure mode. Staleness uses file mtime and last-eval-pass timestamp; thresholds are knobs, not dogma.

## Patterns That Work

**Filing rows point at directories, not individual files.** Directories evolve; filenames don't. A row pointing at `.research/` with a naming pattern survives a hundred new files; a row pointing at `.research/2026-04-21-resolvers.research.md` rots on the next entry.

**Context rows bundle the 2–3 entries a task needs.** A task rarely needs one entry — authoring a hook needs routing + hook best practices + the relevant primitive doc. An entry is either a specific file or a directory to look in; a directory counts as one entry, not N files. Bundling is how the resolver compresses context; single-entry rows leave Claude to assemble the bundle at invocation time.

**Explicit out-of-scope list.** Directories the resolver intentionally ignores (`.raw/`, `.cache/`, `node_modules/`, `.git/`) are listed by name so the auditor doesn't flag them as dark capabilities. Silence is ambiguous; explicit exclusion is load-bearing.

**Naming patterns match directory conventions.** When `.research/` files follow `YYYY-MM-DD-<slug>.research.md`, the filing row encodes that literal pattern. The pattern is discoverable from the directory's existing entries; `/build-resolver` detects it, doesn't invent it.

**Trigger evals with both positive and negative cases.** "X routes to Y" and "X does NOT route to Z" — negative cases catch overlap ("save this research" shouldn't land in `.raw/` just because the text mentions raw data). ~15% negative is a reasonable floor.

**Regeneration over editing.** The managed region is rebuilt from disk, not patched. Regeneration is idempotent; when it produces no diff, the resolver is in sync. A diff on regenerate is the signal that something drifted.

## Anti-Patterns

**Filing table that restates `_index.md` contents.** `_index.md` already describes what a directory holds; the resolver row should link to it, not paraphrase. Paraphrase drifts on the first `_index.md` edit.

**Context entries pointing at vague docs.** `"read the style guide"` without a path is not actionable; the auditor flags it WARN. Every context row names one or more concrete paths that resolve.

**Managed region edited by hand.** The markers exist precisely to prevent this. Hand-edits survive until the next regeneration and then silently vanish. The auditor compares the managed region against a fresh disk scan and flags any row that can't be re-derived.

**Resolver without AGENTS.md pointer.** Unreachable from the root of context. No loading path, no value — a file no session ever opens. The auditor's first check; FAIL on miss.

**Trigger evals absent or untested.** "Tested but static, it rots within 90 days" (Tan) — the claim's magnitude is a T4 anecdote but the direction is consistent with how LLM routing drifts. Evals prove the current state; their absence proves nothing.

**Deep nested resolver hierarchy before the flat one earns its keep.** One resolver at the root handles the vast majority of filing and context decisions. Nested per-plugin resolvers are a scaling option, not a starting shape. Build the root resolver first; add nesting when cross-plugin routing genuinely conflicts.

**Skill-dispatch entries in the filing/context tables.** "When user says 'plan something' → run `/work:plan-work`" belongs in skill descriptions, not the resolver. Conflating dispatch with information routing bloats the resolver and duplicates logic Anthropic already ships.

## Nesting

Nesting is a scaling option: a directory with enough internal filing and context structure to warrant its own routing table gets its own `RESOLVER.md`. Routing follows the **most specific resolver on the filing path** — walking up from the target directory, the nearest `RESOLVER.md` wins. The root resolver covers everything not claimed by a nested one.

**Each resolver owns its evals.** A nested `plugins/build/RESOLVER.md` carries a sibling `plugins/build/.resolver/evals.yml`; the root `.resolver/evals.yml` does not test nested rows. Co-location matches how the auditor operates — one resolver at a time — and keeps nested resolvers self-contained when copied, moved, or extracted.

**The auditor scopes to one resolver.** `/build:check-resolver` takes a target path, walks up to the nearest `RESOLVER.md`, and audits that file plus its sibling `.resolver/evals.yml`. Auditing every resolver in a repo means running the skill once per resolver root; there is no recursive whole-repo mode.

**Don't nest until the flat resolver hurts.** A single root resolver handles most repos indefinitely. Nest only when filing rules genuinely diverge per subtree — a plugin's content really does file differently from the project's — not when the root table feels long. Long is cheap; nested drift is expensive.

## Safety & Maintenance

**Regenerate over edit.** The managed region is a build artifact. `/build:build-resolver --regenerate` rebuilds it from disk; out-of-scope entries and the context table (human-curated) are preserved across regenerations.

**Staleness thresholds are repo-specific.** Defaults: 90 days since last file mtime on `RESOLVER.md` is WARN; 30 days since last eval-pass is WARN. Short-lived repos should lower both; long-dormant docs should raise both.

**Dark capabilities are latent, not urgent.** A directory added to the repo without a filing row is a capability Claude can't reach; the auditor flags it before the first ambiguous file lands in it.

**Evals are public.** `.resolver/evals.yml` is committed to git. Treat failures as you would test failures: the resolver is broken, not the evals.

**No secrets, no PII.** RESOLVER.md is a markdown file committed to git; it inherits normal repo exposure. Never encode paths containing credentials, access tokens, or internal host names as filing or context entries.

---

**Diagnostic when the resolver isn't working:** First check the AGENTS.md pointer — is it present and does it resolve? If not, the resolver is dark. If present, run `/build:check-resolver --run-evals` — failing evals name the routing defect. If evals pass but behavior drifts anyway, the resolver is internally consistent but the AGENTS.md pointer isn't being read; widen or reword the pointer's surrounding context.
